# M6 — SEO and discoverability foundations

> **Status: PLAN — implemented in this same milestone.** Naz's directive (post-M5): no content or copy changes anywhere. Add `sitemap.ts`, `robots.ts`, Article JSON-LD on each archive post, and per-page Open Graph images (no site-wide default).

## Overview

Every page on the site is already discoverable in the browser; this milestone makes them discoverable to search engines and to the link-preview cards that appear when somebody pastes a URL into Slack, iMessage, Bluesky, Mastodon, etc. Three deliverables, in order of "would-be-bad-if-missing":

1. **`/sitemap.xml`** generated by `src/app/sitemap.ts` so search engines can find all 53 routes (homepage, /about, /play-with-me, /archive, plus 49 individual archive posts) with realistic `lastModified` values pulled from each post's frontmatter `publishedAt`.
2. **`/robots.txt`** generated by `src/app/robots.ts` — allows everything, points at the sitemap.
3. **Per-page Open Graph images** so a shared link to `/about` doesn't render with the same image as a shared link to `/archive/14-al-mukit`. Plus **Article JSON-LD** on each archive post so search engines understand each entry as a dated article authored by Sufi Punk.

This milestone deliberately changes **no copy**. Every visible string on every page is unchanged from M5. It only adds metadata, headers, and one tiny invisible `<script type="application/ld+json">` per archive post.

## Goals

1. Ship sitemap + robots in the standard Next.js App-Router-convention way (TypeScript files at `src/app/{sitemap,robots}.ts`, no manual XML).
2. Pull `lastModified` for each archive post directly from frontmatter `publishedAt`. Use the build time as the `lastModified` for `/`, `/about`, `/play-with-me`, `/archive`.
3. For each archive post, render a JSON-LD `Article` block in the page's HTML head/body, with `headline`, `author`, `datePublished`, `description`, `mainEntityOfPage`, and an `image` array.
4. Declare a specific OG image on **every** standard page and on **every** archive post. The chosen images are real photographs already in the project — no AI-generated placeholders, no Unsplash URLs.
5. New `bun run check:seo` lint asserts the metadata shape so a regression (someone deletes the OG image from /about, someone removes the JSON-LD block) is caught immediately.

## Non-goals

- No copy changes — all visible page text is M5-identical.
- No new fonts, no new components, no layout adjustments. The visible rendering of every page is byte-identical to M5.
- No `manifest.json`, no PWA, no service worker. Out of scope for this milestone.
- No Google/Bing Search Console set-up; that's Naz's deploy-side concern.
- No analytics, no Plausible/Fathom, no GA. Same reasoning — privacy-aware site, deploy-time concern, not a code concern.

## 1. Site URL — single source of truth

`src/lib/seo/site.ts` exports:

```ts
export const SITE_URL =
  process.env.NEXT_PUBLIC_SITE_URL?.replace(/\/$/, "") ??
  "https://inspiringthesufi.com";

export function absoluteUrl(path: string): string {
  if (path.startsWith("http://") || path.startsWith("https://")) return path;
  return `${SITE_URL}${path.startsWith("/") ? path : `/${path}`}`;
}
```

The literal `"https://inspiringthesufi.com"` already lives in `src/app/layout.tsx`'s `metadataBase`. M6 replaces that literal with `new URL(SITE_URL)` so the value is set in exactly one place. Vercel preview deploys can override via `NEXT_PUBLIC_SITE_URL` if desired.

## 2. Sitemap

`src/app/sitemap.ts` returns a `MetadataRoute.Sitemap`:

| Route | priority | changeFrequency | lastModified |
|---|---|---|---|
| `/` | 1.0 | `monthly` | build time |
| `/about` | 0.8 | `yearly` | build time |
| `/play-with-me` | 0.8 | `yearly` | build time |
| `/archive` | 0.9 | `monthly` | build time |
| `/archive/[slug]` (×49) | 0.7 | `yearly` | post's `publishedAt` |

`changeFrequency` is per-page-type because:
- The archive itself is finished, so each post's `lastModified` is its original publish date and `changeFrequency: yearly` honestly describes that nothing changes.
- The four standard pages might get small wording tweaks; `monthly` for the index pages, `yearly` for the static read-once pages is honest.
- The homepage gets `monthly` because it embeds the about excerpt (which Naz might edit).

URLs are absolute — sitemap consumers reject relative paths.

## 3. robots.txt

`src/app/robots.ts` returns:

```ts
{
  rules: [{ userAgent: "*", allow: "/", disallow: ["/api/"] }],
  sitemap: `${SITE_URL}/sitemap.xml`,
}
```

The `disallow: ["/api/"]` is defensive — the project doesn't currently expose any `/api/` routes, but the Server Action for `/play-with-me` lives under the App Router's internal action route, and it's good hygiene to keep search bots away from anything action-shaped.

## 4. Open Graph images — one per page, no site-wide default

The current `src/app/layout.tsx` openGraph block does NOT declare an `images: []` array (so there's no inherited default image to remove). M6 adds an `images` array to **every** page's metadata. The mapping:

| Page | OG image | Reasoning |
|---|---|---|
| `/` | `/og/green-mosaic-door.webp` | The literal door this site is traced from. The site's primary visual identity. |
| `/about` | `/og/green-mosaic-door.webp` | About introduces the door — the same image is the right metaphor here. Per-page declaration, not site-wide inheritance. |
| `/play-with-me` | `/threshold/return-to-the-sanctuary.jpg` | The gold-doors threshold maps to Naz's own Play-with-me phrasing: *"as many ways to turn toward the Divine as there are people seeking"* — many doors. |
| `/archive` | `/threshold/return-to-the-sanctuary.jpg` | The same threshold image already appears in the archive's footer; the OG card visually echoes that. |
| `/archive/[slug]` (with hero) | `/archive/{nn}/image-1.{ext}` | Each post's own hero photograph. 25 of 49 posts have one. |
| `/archive/[slug]` (no hero) | `/og/green-mosaic-door.webp` | Quiet site-default fallback for the 24 posts with no hero on file. |

`source-docs/green-mosaic-door.webp` is copied to `public/og/green-mosaic-door.webp` in this milestone so it becomes a real public asset (1.4 MB webp).

## 5. Per-post hero image lookup

`src/lib/seo/post-og.ts` provides:

```ts
export function postOgImage(post: ArchivePost): string {
  // Try post.hero (frontmatter) first — authoritative if Naz set one.
  if (post.hero) return post.hero;
  // Otherwise look on disk for public/archive/{nn}/image-1.{jpeg,jpg,png,webp}.
  // {nn} is the post's slug prefix (e.g. "01" from "01-al-karim").
  // ...
}
```

The `{nn}` prefix is derived from `post.slug` (`01-al-karim` → `01`), because `post.postNumber` can carry slashes (Post 45/46 is one combined entry whose slug is `45-al-zahir-and-al-batin`). The on-disk filename pattern is checked at module import time in a synchronous filesystem read — this all runs server-side at build, so a small `readdirSync` is fine.

If neither a frontmatter `hero` nor an on-disk `image-1.*` is found, the function returns the site fallback `/og/green-mosaic-door.webp`.

## 6. Article JSON-LD

`src/lib/seo/jsonld.ts` exports `articleJsonLd(post)` returning a JS object that gets stringified and rendered as `<script type="application/ld+json" dangerouslySetInnerHTML={{ __html: JSON.stringify(...) }} />`.

Shape:

```jsonc
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "Al Karim — The Generous",
  "author": { "@type": "Person", "name": "Sufi Punk" },
  "datePublished": "2015-12-08",
  "description": "Al Karim (The Generous) — Man in the Mirror, Michael Jackson",
  "mainEntityOfPage": "https://inspiringthesufi.com/archive/01-al-karim",
  "image": ["https://inspiringthesufi.com/archive/01/image-1.jpeg"],
  "publisher": {
    "@type": "Organization",
    "name": "Inspiring the Sufi",
    "url": "https://inspiringthesufi.com"
  }
}
```

For the dual-Name post (Post 45/46), `headline` becomes `"Al Zahir & Al Batin — The Manifest & The Hidden"` so the structured data preserves the dual-Name reality.

`author.name` is `"Sufi Punk"` per Naz's directive. The `publisher` block is filled with `Inspiring the Sufi` to indicate the site itself, since Sufi Punk the person ↔ Inspiring the Sufi the site is the dual relationship Naz operates under.

## 11. Edge cases / failure modes

| Case | Behaviour |
|---|---|
| Post 45/46 dual-Name post | `headline` includes both Names joined with `&`. Image lookup uses the slug prefix. |
| Post with no on-disk hero | `image` array contains the site default `/og/green-mosaic-door.webp`. |
| Post hero with non-JPEG extension (e.g. `.png`, `.webp`) | `post-og.ts` checks `.jpeg`, `.jpg`, `.png`, `.webp` in that order. First match wins. |
| `NEXT_PUBLIC_SITE_URL` set to `https://staging.example.com/` (with trailing slash) | Slash is stripped by the SITE_URL definition. |
| `process.env.NEXT_PUBLIC_SITE_URL` not set | Falls back to `https://inspiringthesufi.com`. |
| New post added later | `sitemap.ts` reads from the same `loadAllArchivePosts()` source the routes use, so it picks up automatically. The lint catches sitemap/post-count drift. |
| Naz removes a post hero from disk | OG falls back to site default. Lint reports the change in hero-coverage count. |

## 12. Implementation notes

- Both `sitemap.ts` and `robots.ts` are server modules; they import `loadAllArchivePosts` from `@/lib/archive/loader` (which is already `"server-only"`).
- The JSON-LD block is rendered in the body of `/archive/[slug]/page.tsx`, immediately after `<SiteHeader />`. This is valid and is the same place every Next.js JSON-LD example places it.
- All metadata changes use the App Router's `Metadata` / `generateMetadata` APIs; no manual `<meta>` tags are written.
- The `/archive/page.tsx` is currently a server component with a `searchParams` prop, so it can't define a static `metadata` constant — it gets `generateMetadata` instead.
